# OpenClaw Runtime 阶段1风险评估报告

## 1. 风险清单（按严重程度）

| 严重度 | 风险 | 触发条件 | 影响 |
|---|---|---|---|
| 高 | 依赖调度错误导致死锁/饥饿 | DAG 计算错误或状态不一致 | 全链路阻塞 |
| 高 | 跨部门数据一致性问题 | Mongo 成功、Redis 失败或反之 | 误判依赖就绪 |
| 高 | 外部 skill 波动放大失败 | 网络抖动/限流/超时 | 批量失败 |
| 中 | 结果聚合偏差 | 部门输出 schema 不一致 | 大脑融合失真 |
| 中 | 并发争用导致性能退化 | 高并发任务同时执行 | 延迟显著增加 |
| 中 | 兼容性回归 | 改造破坏旧执行路径 | 老接口不可用 |
| 低 | 监控噪音过高 | 指标定义粗糙 | 告警疲劳 |

## 2. 缓解措施

### 2.1 调度与一致性
- DAG 调度前做环检测；任务启动前做依赖快照。
- 引入状态机版本号（乐观并发控制）。
- 关键写操作采用“先 Mongo 后 Redis + 失败补偿重放”。

### 2.2 外部调用稳定性
- Skill 调用超时上限（例如 8s/15s 双阈值）。
- 指数退避重试（最多 2~3 次）。
- 熔断与半开探测，避免雪崩。

### 2.3 兼容与回滚
- 双执行路径：`legacyMode` 与 `orchestratedMode` 并存。
- 灰度开关：按任务类型/租户开启。
- 回滚策略：一键切回 legacy executor。

## 3. 应急预案

## 3.1 P0 故障应急
1. 立即切换 `legacyMode=true`。
2. 停止新任务调度，仅允许已在途任务完成。
3. 导出失败任务列表，进入补偿队列。
4. 启动只读监控模式，排查根因。

## 3.2 数据修复
- 根据 Mongo 的 `department_outputs` 回放 Redis 状态。
- 使用 `taskId + department + timestamp` 做幂等修复。

## 4. 监控指标定义

| 指标 | 含义 | 阈值建议 |
|---|---|---|
| `task_e2e_latency_ms` | 任务端到端耗时 | P95 < 20s |
| `department_success_rate` | 部门执行成功率 | > 98% |
| `skill_timeout_rate` | skill 超时率 | < 3% |
| `dependency_wait_ms` | 依赖等待时长 | P95 < 5s |
| `aggregation_fail_rate` | 聚合失败率 | < 1% |

## 5. 风险验收门槛

- 连续 3 天压测下 P95 延迟与成功率达标。
- 回滚演练至少 1 次并通过。
- 任一部门故障时，系统可降级而非整体崩溃。
