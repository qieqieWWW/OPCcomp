# OPC 比赛演示版后续计划（阶段目标锁定）

更新时间：2026-04-05
阶段目标：先做可演示、可执行、可控风险的版本用于参赛。

---

## 0. 本阶段总目标

在当前 OPC 基础上，优先完成 3 项能力：

1. 审批闸门（高风险动作强制拦截）
2. 真执行样板 A（market）：通过技能链路操作即梦网站生成可用 AI 视频
3. 真执行样板 B（sales）：接入用户界面的 AI 客服应答，帮助老板处理客户咨询

说明：本阶段以“能跑通演示闭环”为第一优先，不追求一次性全量工程化。

---

## 1. 任务一：审批闸门（必须先做）

### 1.1 目标
将“审批”从 prompt 约束升级为 runtime 强制机制：
- 未审批的高风险动作不能执行
- 审批结果可追踪、可回放

### 1.2 首批拦截范围（高风险）
- 支付与扣费行为
- 外部账号敏感操作（实名、资质提交、关键权限变更）
- 不可逆动作（批量发布、主配置覆盖、删除类动作）
- 本阶段新增：即梦视频生成相关的正式扣费动作

### 1.3 交付结果
- 审批状态机：`pending -> approved/rejected -> executed`
- 审批事件写入执行轨迹（trace）
- 前端/演示端可看到“待审批动作”与“审批结果”

### 1.4 验收标准
- 任一高风险动作在未审批时，执行器返回阻断状态
- 审批通过后可继续执行，审批拒绝则中止并返回原因
- 演示中可展示完整审批链路证据

---

## 2. 任务二：真执行样板 A（market）

### 2.1 场景定义
输入市场传播计划后，market 部门通过技能链路操作即梦网站（https://jimeng.jianying.com/ai-tool/home/?utm_medium=baiduads&utm_source=pinzhuan&utm_campaign=title）产出可用 AI 视频素材。

### 2.2 前置条件
- 老板提供可用的即梦账号登录态（或可由技能系统复用的浏览器会话）
- 明确最小输入参数：主题、风格、时长、分辨率、文案/脚本

### 2.3 执行闭环（演示版）
1. market 输出视频生成任务参数
2. 触发审批闸门（高风险：扣费生成）
3. 审批通过后通过浏览器技能访问即梦站点并执行生成
4. 获取视频结果（链接/文件ID/元数据）
5. 结果写回黑板并进入最终聚合

### 2.4 交付结果
- market 真执行能力不再是 mock
- 可生成至少 1 条可播放的 AI 视频结果
- 结果可在演示输出中展示（含回执信息）

### 2.5 验收标准
- 在同一任务链中看到：计划 -> 审批 -> 调用 -> 回执 -> 聚合
- 失败时可返回可读错误（鉴权失败/余额不足/超时）
- 保证一次任务不会重复扣费（最小幂等保护）

---

## 3. 任务三：真执行样板 B（sales AI 客服）

### 3.1 场景定义
将销售专家能力中的“AI 客服应答”接入用户界面，支持老板把咨询交给系统处理。

### 3.2 最小能力边界（演示版）
- 支持单轮/短多轮咨询应答
- 识别咨询意图（产品介绍、价格咨询、预约演示）
- 输出建议动作（如：登记线索、安排跟进）

### 3.3 执行闭环（演示版）
1. 用户在 UI 输入咨询问题
2. sales AI 生成客服回复
3. 返回可读回复 + 动作建议
4. 可选写回 CRM/线索记录（若未接 CRM，先本地记录）

### 3.4 交付结果
- UI 中真实可交互的客服应答能力
- sales 不再仅输出静态模板结果
- 能展示“老板咨询 -> AI 回复 -> 下一步动作”

### 3.5 验收标准
- 至少覆盖 3 类常见咨询（产品、价格、试用/演示）
- 回复可读、不过度承诺、符合审批/合规约束
- 演示可稳定重复（不依赖手工改代码）

---

## 4. 推荐执行顺序（严格）

1. 审批闸门
2. market 即梦真执行
3. sales UI 客服应答

原因：后两项都会涉及真实动作与风险，必须先有可执行的审批拦截。

---

## 5. 演示版本 Definition of Done

当以下条件同时满足，视为本阶段完成：

- 有审批闸门，且高风险动作不能绕过
- market 可在审批后生成真实 AI 视频并回写结果
- sales 可在 UI 中进行客服应答并输出动作建议
- 可完成一次端到端演示：
  老板下达目标 -> 系统计划 -> 审批 -> 执行 -> 回执 -> 展示

- 新需求进入时，先判断是否影响上述三项主线。
- 若出现冲突，以“先保障演示闭环”优先。
